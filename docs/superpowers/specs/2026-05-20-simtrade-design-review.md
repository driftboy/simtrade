## 三、具体优化建议

### 3.1 补充权限系统设计

```
# RBAC 权限模型
User → Role → Permission

Permission 结构：
- resource: 资源类型 (transaction/document/user...)
- action: 操作类型 (create/read/update/delete/approve...)
- scope: 作用域 (self/class/all)
```

### 3.2 补充单证模板字段配置

```json
{
  "doc_type": "commercial_invoice",
  "fields": [
    {
      "name": "invoice_no",
      "type": "text",
      "required": true,
      "validation": "^[A-Z0-9]{6,20}$"
    },
    {
      "name": "invoice_date",
      "type": "date",
      "required": true,
      "default": "{{today}}"
    }
  ]
}
```

### 3.3 补充 REST API 设计

| 资源 | GET | POST | PUT | DELETE |
|------|-----|------|-----|--------|
| /api/transactions | list | create | update | delete |
| /api/documents | list | create | update | delete |
| /api/courses/{id}/scores | list | - | - | - |

### 3.4 补充数据字典

### 3.5 补充缓存策略

```
缓存层次：
1. 浏览器缓存：静态资源
2. CDN 缓存：图片、CSS、JS
3. 应用缓存：商品、汇率、配置等基础数据
4. 数据库缓存：查询结果

缓存失效策略：
- 商品数据：1小时
- 汇率数据：配置变更时
- 用户会话：30分钟
```

### 3.6 补充错误码设计

```
# 错误码格式: {模块}{类型}{代码}
# 模块: 01=用户, 02=交易, 03=单证...
# 类型: 1=业务错误, 2=校验错误, 3=系统错误

0101: 用户不存在
0102: 密码错误
0201: 交易不存在
0202: 交易状态不允许此操作
0301: 单证审核不通过
0302: 单证日期逻辑错误
```

## 四、架构优化建议

NotificationService

## 五、安全加固建议

### 5.1 认证安全

- 密码复杂度要求
- 登录失败锁定
- 会话超时控制
- CSRF 防护

### 5.2 数据安全

- 敏感数据加密存储
- SQL 注入防护 (Django ORM)
- XSS 防护 (模板转义)
- 文件上传校验

### 5.3 业务安全

- 权限校验
- 操作审计
- 数据隔离 (课程级)
